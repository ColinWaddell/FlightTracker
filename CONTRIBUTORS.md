# Contributing

Thanks for taking an interest in FlightTracker.

FlightTracker is a personal hobby project that I work on because I genuinely enjoy building it. It's grown far beyond what I originally expected, and I'm always pleased to see other people building one for themselves, suggesting ideas, reporting bugs, and occasionally contributing code.

This document isn't intended to put people off contributing. It's simply here to explain how I maintain the project, what I'm looking for when people contribute, and what you can expect in return.

## Feature requests

Ideas are always welcome.

If you think FlightTracker is missing something, open an issue and tell me about it. Even if I don't implement it straight away, it's useful to know how people are using the project and what they'd like it to do.

## Pull requests

Pull requests are welcome, and I do merge them.

That said, I'm fairly opinionated about the direction of the project. A pull request being technically correct doesn't necessarily mean it's something I want to merge. Sometimes I'll choose to close a PR simply because I'd rather implement the idea myself.

Please don't take that personally. It's just how I enjoy maintaining the project.

## AI-generated code

I'm not opposed to AI-assisted development.

If you submit AI-generated code, I expect you've reviewed it thoroughly, understand every line, and are happy to stand behind it as if you'd written it yourself.

I'll naturally have more confidence in contributions from developers with an established track record. If your account is very new, has little public history, or your contribution suggests you aren't yet comfortable with the codebase, I'm much more likely to ask you to develop the idea further before merging it. The same applies to handwritten code as well—this isn't about AI, it's about confidence in the contribution.

## Why I'm selective

FlightTracker isn't a project that needs a large development team. It's intentionally small enough that I can comfortably maintain it myself, and I enjoy doing that.

That means I'm often more interested in discussing ideas than accepting implementations.

## Issues and support

If you've found a genuine bug, please open a GitHub issue. That's by far the best place to report problems, track progress, and make sure nothing gets forgotten.

I'm always happy to chat elsewhere, but conversations on Discord, Reddit, Mastodon or other platforms are easy to lose. If it's something that needs fixing, please put it on GitHub.

I also have a full-time job, a family, and two young kids. I'll do my best to respond to bug reports and support requests, but FlightTracker is a hobby project. Unless something is seriously broken and affecting lots of people, please don't expect an immediate response.

Thanks for understanding, and thanks again for helping make FlightTracker better.
